# Promco x Saleor Architecture: Backend-First Composition

## Core Philosophy
**"Frontend gets data in a single call. Backend owns the complexity."**

We avoid the anti-pattern of the frontend calling Saleor directly. 
Instead, the frontend calls our **Promco API**, which acts as an orchestration layer.

### The Pattern
```js
// Frontend calls
const { data } = await api.products.list();

// backend service (packages/services/src/product.service.ts):
async function listProducts(orgId: string) {
  // 1. Fetch from Saleor (Catalog Source of Truth)
  const saleorProducts = await saleor.products.list({ channel: orgId });
  
  // 2. Fetch FIFO costs from your DB (Profit Source of Truth)
  const costs = await db.fifoBatches.getByCosts(productIds);
  
  // 3. Compose into enriched response
  return saleorProducts.map(p => ({
    ...p,
    currentFIFOCost: costs[p.id]?.unitCost,
    profitMargin: calculateMargin(p.price, costs[p.id]?.unitCost),
    isProfitable: p.price > costs[p.id]?.unitCost
  }));
}
```

## Architecture Layers

### 1. `packages/coremmerce` (The Integration Layer)
This package is the **only** place that talks to Saleor directly. It abstracts the GraphQL complexity.

*   **Tech Stack**: `graphql-request` (Lightweight, Promise-based).
*   **Responsibility**:
    *   Execute GraphQL queries/mutations against Saleor.
    *   Handle Saleor Authentication (App Token).
    *   Transform raw GraphQL errors into typed Result objects.
    *   **NEVER** contains business logic. It just fetches and returns.

### 2. `packages/services` (The Business Logic Layer)
This is where the magic happens. Services consume `coremmerce` and `@repo/db`.

*   **Responsibility**:
    *   **Composition**: Merge Saleor data with Promco DB data.
    *   **Profit Logic**: Calculate margins, FIFO, etc.
    *   **Orchestration**: If a user updates a product, update Saleor first, then update local DB if needed.

### 3. `apps/web/api` (The Exposure Layer)
REST API endpoints (Next.js Route Handlers) that call `packages/services`.

*   **Responsibility**:
    *   AuthN/AuthZ (Validate User Session).
    *   Input Validation (Zod).
    *   Call Service.
    *   Return JSON.

## Key Decisions

### Why `graphql-request`?
*   We are moving logic to the backend.
*   We don't need the heavy caching/state management of `urql` on the server-side for simple service calls.
*   It's simple, typed, and works well with `graphql-codegen`.

### Data Ownership
*   **Saleor Owns**:
    *   Product Catalog (Name, Description, Images, Variants)
    *   Sales Price (The price the customer pays)
    *   Orders (The transaction record)
    *   Customers (Basic profile)
*   **Promco Owns**:
    *   **Cost Price** (FIFO Batches) - Saleor has NO concept of this.
    *   **Profit Calculations**
    *   **Inventory Reconciliation Events**
    *   **Campaign Profitability Logic**

## Implementation Guide

### 1. Setup Coremmerce Client
In `packages/coremmerce/src/client.ts`:
```js
import { GraphQLClient } from 'graphql-request';
import { getSdk } from './graphql/generated/sdk'; // Generated by codegen

const client = new GraphQLClient(process.env.SALEOR_API_URL!, {
  headers: {
    'Authorization': `Bearer ${process.env.SALEOR_APP_TOKEN}`
  }
});

export const saleor = getSdk(client);
```

### 2. Write a Service
In `packages/services/src/product.service.ts`:
```js
import { saleor } from '@repo/coremmerce';
import { db } from '@repo/db';

export const productService = {
  async getEnrichedProduct(id: string) {
    // Parallel fetch for performance
    const [saleorData, localData] = await Promise.all([
      saleor.ProductDetails({ id }),
      db.query.productCosts.findFirst({ where: eq(productCosts.saleorId, id) })
    ]);
    
    return {
      ...saleorData.product,
      fifoCost: localData.cost,
      // ... composition
    };
  }
}
```

## Error Handling
*   If Saleor is down -> Service throws a `503 Service Unavailable`.
*   If Saleor returns GraphQL Error -> `coremmerce` wraps it in a typed Error, Service catches it and decides (retry or fail).

## Webhooks (The "Back" Channel)
When something changes in Saleor (e.g., Order Created), Saleor calls Promco.
*   **Endpoint**: `/api/webhooks/saleor`
*   **Logic**:
    1.  Verify Signature.
    2.  Identify Event (`ORDER_CREATED`).
    3.  Call `OrderService.processNewOrder(payload)`.
    4.  `OrderService` calculates profit for that order and saves to Promco DB.
